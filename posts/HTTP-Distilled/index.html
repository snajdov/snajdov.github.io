HTTP Distilled

snajdov.me
2020-04-14 (Updated: 2020-11-12)
networking › protocols › http
client, http, server

NOTE: HTTP version 2 (HTTP/2) works differently. For HTTP/2 see: HTTP/2 Distilled (work in progress…)

Every time we use Google to search for something, or play our favorite song on Youtube, we are using HTTP. It stands for HyperText Transfer Protocol, and it powers the whole World Wide Web.

Basics

Let’s say we typed “www.google.com" in our browser. When we hit enter, something happens, and we’re presented the Google homepage.
So, what does happen?

More or less, the following:

  1. Our browser (Firefox, Chrome, …), acting as a Client, generates a new HTTP Request.
  2. That request, using TCP is sent to the Server.
  3. The Server reads the request, does some processing, generates a result, and stores it in an HTTP Response.
  4. That response is sent back to the Client using TCP.
  5. The Client (our browser) reads the response, and generates the view we are seeing.

From above, we can identify two participants in the communication:

  1. The Client: requests resources
  2. The Server: serves resources

as well as, two types of HTTP messages:

  1. HTTP Request: contains information on the resource the client wants
  2. HTTP Response: contains the resource returned from the server

Verbs / Methods

We need a mechanism to tell the server our intentions. Do we want to get something from the server, or do we want to upload something to it? Maybe edit/delete something? For that, there is a reserved HTTP Verb or Method field that every HTTP Request starts with.
These are words that define the type of Client action. It can be one of:

  • GET: Retrieve a resource
  • POST: Upload to a resource
  • PUT: Update a resource
  • PATCH: Partially update a resource
  • DELETE: Delete a resource
  • HEAD: Get only meta information (GET with no body)
  • OPTIONS: Check the communication options of the server
  • TRACE: Perform a message loop-back test
  • CONNECT: Establish a connection with the specified resource

More on these, here: https://developer.mozilla.org/en-US/docs/Web/HTTP/Methods

Next, we define the Resource we want to act upon. Typically, the resource is specified in the form: “/Path/To/Resource”. This is similar to navigating a file-system in Linux. Headers

Now we come to the interesting part: Headers.

Up until know, we learned how to specify our actions (verb/method), the resource we want to act upon (resource), apply additional filters to the resource (query string), and even specify how to show the returned data (fragment).
So, what is there left to do? A lot!

Headers are case insensitive key/value pairs. With the Headers we talk directly to the Server. We can say to it:

  • I’m a Firefox user
    User-Agent: Firefox (this is simplified, it’s a lot longer)
  • I can understand HTML and JSON formats
    Accept: text/html,application/json;q=0.9
  • I am sending you JSON formated data
    Content-Type: application/json
  • The website I request is snajdov.me
    Host: snajdov.me
    NOTE: one server can server many websites, i.e. domains
  • I don’t want you to track me
    DNT: 1
  • I have some information you gave me, from the last time we communicated
    Cookie: X=some-cookie-data
  • Don’t close this TCP connection
    Connection: keep-alive
  • and many more…
    https://developer.mozilla.org/en-US/docs/Web/HTTP/Headers

As we can see, the posibilities are endless. Headers give meaning to the communicated data, and change the behaviour of the applications. We can define our own custom headers for our applications, for an even richer experience.

Body

Lastly, we are given the opportunity to send something to the Server. The Body is where all our data goes. It is anything after the blank line. Usually the Body is only present in POSTs, PUTs and PATCHes, since the other Methods are not meant for uploading data.
NOTE: By usually I mean: you can send the Body even in GET, but it’s up to the Server to decide whether it’ll parse the body.

Now the Server has all the info needed to process this request, and do its job. It can fetch files, store information in databases, or communicate with other servers. When everything’s done, it can start creating the HTTP Response.

Take a breath, we’re going back! :)

Status Codes

It’s obvious that not every request can be successfully served. We may send the wrong stuff or the wrong format, the requested resource may be missing, the server may have lost connection to other systems, or other systems will take over this connection. Because of this, status codes are divided into five classes. Here are some of the often used status codes, and their meanings:

  1. 1XX - Informational responses
    • 101 SWTICHING PROTOCOLS
    • …
  2. 2XX - Successful responses
    • 200 OK
    • 201 CREATED
    • 202 ACCEPTED
    • …
  3. 3XX - Redirects
    • 301 PERMANENT REDIRECT
    • 302 FOUND
    • …
  4. 4XX - Client errors
    • 400 BAD REQUEST
    • 401 UNAUTHORIZED
    • 403 FORBIDDEN
    • 404 NOT FOUND
    • …
  5. 5XX - Server errors
    • 500 INTERNAL SERVER ERROR
    • 502 BAD GATWAY
    • 503 SERVICE UNAVAILABLE
    • …

More on this, here: https://developer.mozilla.org/en-US/docs/Web/HTTP/Status

When returned to the Client, these status codes tell us what happened on the Server-side. Depending on the code, we should act accordingly.
Despite the statuses, everything else in the Response is the same as the Request.

And with that, the whole communication is complete! :)

Example: Inspecting HTTP using Wireshark

With the previous app started, we can trace the communication between the browser and the server using Wireshark.
Once you start Wireshark, listen on the Loopback: lo interface, and access http://localhost:8080/hi on your browser.
NOTE: If you have permission issues, run Wireshark as root.

Once you get the response, in Wireshark, in the filter input, write: tcp.port == 8080 to remove all other packets.
You should get something like this:

It shows the following (order by column No.):

  1. [SYN] Initiation of a TCP connection from the Browser (port 55448), to the Server (port 8080)
    (in direction Client->Server)
  2. [SYN, ACK] Server accepts the connection, and initiates a connection to the Client
    (in direction Server->Client)
  3. [ACK] The Client accepts the Server connection
    (Now we have a full-duplex connection with the server)
  4. [HTTP GET] The Client sends the HTTP Request
    (you can see the full request on the screenshot)
  5. [ACK] The Server-side TCP accepts this segment
    (After this, the server processes the request)
  6. [HTTP 200 OK] Once processed, the Server returns an HTTP Response
  7. [ACK] The Client-side TCP accepts this segment
    (After this, the browser gets the response, and renders the view)
  8. [FIN, ACK] The Client closes the TCP connection for sending
    (in direction Client->Server)
  9. [FIN, ACK] The Server accepts this closing, and the Server closes the connection for sending
    (in direction Server->Client)
  10. [ACK] The Client accepts this closing

Of particular importance are No. 4 and 6. This is where the HTTP Request (no. 4) and the HTTP Response (no. 6) are sent using TCP.

TL;DR

Clients (usually Browsers) send HTTP Requests specifying the action (method), the resource (possibly with query string and fragment) and other metadata (Headers) to the Server. The Server parses this and computes a result, which gets stored in an HTTP Response containing a status code, headers and body that is sent back to the Client. Header metadata is used to better understand each other.
