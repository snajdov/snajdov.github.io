Basics of Information Security

snajdov.me
2020-04-16 (Updated: 2020-11-16)
security › cryptography
authenticity, confidentiality, decryption, encryption, information, integrity, mac, security

When talking about information security, people usually think about two things: cryptography, and hacking.

Although, this is not far from the truth, it is not exactly the same. Let me explain.

Introduction

First, let’s make a few things clear:

  • What is “data”, and what is “information”?
  • Do they differ? How?

It’s key to understand these fundamental concepts, because they can make understanding the later concepts easier.

OK, so, first “data”. What is data?
In computing terms, we can say that data is:

Any sequence of 1s and 0s.
ANY.
It doesn’t matter the order, the frequency or anything.
If you have bits, you have data.

Alright, what about “information”?
Let’s think about this: “INFOrmation”. Information means:

Extracting some knowledge from the data.
It’s understanding the data.

This is important!
We can have data, but unless we understand it, it meaningless, i.e. we don’t get any information.

Right. So, with this new reasoning about information:
What is “Information Security”?

Obviously, securing information, but now we can say:
Securing the understanding of data. Securing the knowledge in the data.
Note the “understanding” part. We don’t secure the data, we secure the “understanding of data”.

OK, what is there to secure about the “understanding” of data?
There are generally 3 key properties of any Information, collectively called CIA (no, not that CIA):
(1) Confidentiality,
(2) Integrity, and
(3) Availability

(1) The first, is the most popular one: Confidentiality, it means:

The INABILITY of someone to extract the INFORMATION from the DATA, WHILE IN TRANSIT FROM SENDER TO RECEIVER, if the data was NOT meant for him/her.

Let’s think about this. A third party (some intruder) CAN HAVE THE DATA, but CAN NOT HAVE THE INFORMATION.
That is why it’s important to differentiate these two things.
Note that, Confidentiality also entails that the corresponding party (the receiver) is ABLE TO EXTRACT THE INFORMATION FROM THE DATA.
But, ideally, ONLY him/her.

(2) OK. Next we have Integrity. In this context it means:

Ensuring that the INFORMATION HAS NOT CHANGED WHILE IN TRANSIT FROM SENDER TO RECEIVER.
NOT ONE BIT.

Note that, this property DOES NOT mean the information is confidential.
Its ONLY job is to secure the IMMUTABILITY of information.

(3) Finally, Availability is:

The ability to ACCESS information ANYTIME from ANYWHERE.

This one is a little bit odd, because usually security is involved with denying access.
But, look at it this way, it’s:

Ensuring the ACCESS of information to the PERMITTED USERS, despite the CONSTANT ATTACKS TRYING TO DENY IT.

You see, breaking security isn’t all about hacking into the system.
Denying a service by a DDoS is also a SECURITY RISK.

Usually people confuse these, and mix them up. So I’ll put it in quotes just so it’ll stick better:

These concepts are INDIVIDUAL.
If you want ALL, you need to PROVIDE ALL, through different mechanisms.

We DO NOT ALWAYS need ALL of them, or maybe not to the absolute degree.
We’ll see later on how we can combine them to achieve different things.

So, these are the PROPERTIES of Information.
Now we’ll see the MECHANISMS used to ENSURE THEM.

Following is a short table showing the mechanism for each property:

Property Mechanism
Confidentiality Cryptography
Integrity Message Authentication Codes (MAC)
Availability Replication/and more…

Each mechanism is described in the following subsections.

Confidentiality Through Cryptography

How do we achieve Confidentiality? Using Cryptography.
OK, let’s think about this too. Cryptography allows us to fullfill the Confidentiality property of an Information.
With that Cryptography is just one subpart of Information Security.
We’ll talk A LOT about cryptography, but remember that it ONLY SOLVES CONFIDENTIALITY.
Nothing else.

Now, cryptography is a large topic for discussion, too large for this post, but we’ll cover the basic concepts.
Although cryptography is quite older than computers, we’ll talk in terms of its usage in computers.

Also, before we delve into the concrete specifics, we should mention the following terms:

  • PLAIN TEXT: information readable by anyone (anyone has the DATA + INFORMATION)
  • CIPHER TEXT: information readable only by the one with the decryption key (anyone has the garbled up DATA, but ONLY WITH THE KEY we have INFORMATION)
  • ENCRYPTION: the transformation of PLAIN TEXT, to CIPHER TEXT
  • DECRYPTION: the transformation of CIPHER TEXT, to PLAIN TEXT
  • KEY: a shared secret between the sender and receiver (usually a random HUGE number, like 2^256 huge, or more)
  • CIPHER: the algorithm used to ENCRYPT/DECRYPT the data
  • MODE: the encryption mode used to CHAIN different BLOCKS in the ENCRYPTION/DECRYPTION
  • BLOCK: a block/range of the data to be ENCRYPTED/DECRYPTED

Using these, we can categorize cryptography whether it uses blocks:
BLOCK vs STREAM CIPHERS
and whether it uses one or more keys:
SYMETRIC VS ASYMETRIC

The usual workflow is this:
(1) We have some PLAIN TEXT that we want to send to someone
(2) Using the KEY, we ENCRYPT the PLAIN TEXT and get the CIPHER TEXT
(3) We send the CIPHER TEXT through the channel
(4) The receiver and all other listening get the CIPHER TEXT
(5) ONLY the receiver can DECRYPT it using the KEY to get the PLAIN TEXT

A couple of things to consider:

  • Is the ENCRYPTION secure?
  • Do we use the SAME KEY for DECRYPTION?
  • How do we send the KEY to the receiver without anyone else getting it?

Let’s discuss these one by one.

Is the ENCRYPTION secure? (in the following text, we I say Encryption, by default assume Decryption as well)
There is no easy answer. Testing whether the encryption is secure is no easy task.
It’s best if we can mathematically express its randomness.
But first, let’s decompose ENCRYPTION:

ENCRYPTION has two parts:

  • CIPHER: the ALGORITHM used to ENCRYPT the PLAIN TEXT
  • MODE: (if it’s a BLOCK CIPHER) the way we CHAIN BLOCKS when ENCRYPTING/DECRYPTING
